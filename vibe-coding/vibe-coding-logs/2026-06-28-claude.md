### Recommended approach

For a GTK app, I recommend a **layered architecture that's MVC-flavoured but pragmatic**, rather than a strict three-tier or textbook MVC. Here's why and how:

The natural seams in your code already are:
- **Constants/config** — names, extensions, sort modes, the `ALL_NOTES` sentinel.
- **Model (data layer)** — `Note`, `collect_notes`, `immediate_subfolders`, plus the file read/write logic currently buried in the window. This layer is pure Python with zero GTK imports, which makes it trivially testable.
- **Syntax highlighting** — `MarkdownHighlighter`. It touches GTK (`TextBuffer`) but is self-contained; it's a view-helper.
- **View/Controller (the window)** — `NotebookWindow`. In GTK, the view and controller are hard to fully separate because signal handlers (controller logic) are wired directly to widgets (view). Fighting this produces awkward indirection. So I'll keep them together but lean on the model layer for all disk/business logic.

I'd avoid strict three-tier (presentation/logic/data as rigidly isolated tiers) here because there's no service tier worth abstracting — it's a local file editor. MVC adapted to GTK's reality is the better fit.

Proposed layout:

```
qdvc_markdown_notebook.py          # thin entry point: argv parsing + main()
qdvcmdnb_lib/
    __init__.py
    config.py                      # constants, APP_NAME, sort modes, ALL_NOTES
    model.py                       # Note, collect_notes, immediate_subfolders, file I/O
    highlighter.py                 # MarkdownHighlighter
    window.py                      # NotebookWindow (view + controller)
```

One refinement to the model: I'll move the file read/write out of the window into the model layer (a `NoteRepository` or simple module functions), so the window never touches disk directly. That's the main genuine improvement beyond just splitting files.
